# Star-Growth Pattern Checklist

Traits that correlate with repos gaining stars fast. Use as a scoring rubric against candidate niches — more checks isn't a guarantee, but it shifts odds.

## Framing patterns
- **"Self-hosted alternative to [popular SaaS]"** — people search "self-hosted X" constantly; instant positioning.
- **"[Boring infra] but for AI agents"** — retrofitting a known dev-tool category (queues, auth, vector DB, observability) for the current AI-agent wave.
- **Single powerful primitive, zero-config** — "one command, it just works" beats configurable-but-complex on launch day.
- **Wrapper/glue for a hot API or model the day it drops** — e.g. day-one CLI/SDK wrapper for a newly released model or API. Speed matters more than polish here.
- **Local-first / privacy angle** — "runs entirely on your machine, no data leaves" resonates on HN/Reddit.
- **Comparison bait** — repo positions itself directly against a well-known incumbent (README has a comparison table) — gives people a reason to link/share it.

## Presentation patterns
- **Killer demo GIF/video in first screen of README** — single highest correlate with HN front-page repos.
- **Install-to-result in under 60 seconds** — copy-pasteable one-liner, no account/API key friction for the first demo.
- **Name is short, guessable, greppable** — avoid generic names that collide with existing popular repos.
- **Badges + clean structure signal "maintained"** — even pre-1.0, a tidy README reads as trustworthy.

## Timing/channel patterns
- **Show HN on Tue-Thu morning US time** tends to get more traction than weekends.
- **Piggyback on a fresh news hook** — a new model/framework/regulation drop creates a window where "first tool for X" gets outsized attention for ~1-2 weeks.
- **Cross-post the same launch** to Show HN + relevant subreddit + Product Hunt same day — compounding discovery, not sequential.

## Red flags (deprioritize)
- Niche already has 3+ repos with 1k+ stars launched in the last month — crowded, marginal returns.
- Requires a paid API key or signup before the user can see any output — kills casual GitHub browsers.
- No visual/demo-able output (e.g. a backend library with no UI) — harder to make shareable, though still viable for infra niches with strong technical differentiation.

## How to score a candidate niche
Count how many "framing" + "presentation" traits realistically apply, note timing window (is there a live news hook right now?), and flag competition level from live research. Present this as a short rationale next to each shortlist item, not a raw score out of N — the reasoning matters more than the number.
